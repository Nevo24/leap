# Editor-GUI agent rows: Cursor Agent tabs + VS Code Copilot chats (read-only monitor rows)

Optional, **on by default** (Settings -> "Show editor Agent sessions (Cursor + VS Code Copilot)", persisted as `show_editor_gui_agents` in `monitor_prefs.json`; a legacy `show_cursor_gui_agents` value is migrated once at startup in `MonitorWindow.__init__`. All default reads - the `SessionRefreshWorker` gate + the two settings-dialog plumbing reads - default to `True`, so a user who never touches the toggle gets the rows; each scan is a cheap no-op when its editor isn't running). When on, `SessionRefreshWorker` calls `scan_open_cursor_agents()` + `scan_open_vscode_copilot_sessions()` each refresh tick (independent try/except - one editor's schema drift can't take the other's rows down) and the monitor shows one row per **open Cursor Agent/Composer tab** and one per **recently-active VS Code Copilot Chat session**. These are NOT Leap sessions - they are a pure display overlay (no PTY, no server, no queueing).

**Shared plumbing.** Cursor rows: `row_type == 'cursor_agent_gui'` (`CURSOR_GUI_ROW_TYPE`), tag `cursor-gui:<composerId>`. VS Code rows: `row_type == 'vscode_copilot_gui'` (`VSCODE_GUI_ROW_TYPE`), tag `vscode-gui:<sessionId>`. The combined tuples `GUI_ROW_TYPES` / `GUI_TAG_PREFIXES` live in `vscode_copilot_scan.py` and are what every editor-agnostic code path keys on (`str.startswith` takes the tuple directly): render dispatch to `_build_editor_gui_row`, reconcile/synthesis, drag-reorder, alias/row-color orphan pruning, status-fire skip, settings-off teardown, SCM provider-reconnect skip. Both scans emit the same row shape with the generic keys `window_folder` / `chat_id` / `chat_name` (renamed from the old `cursor_window_folder` / `composer_id` / `composer_name`).

**Two dedup seams keep the per-editor code from drifting.** (1) **Open-workspace detection** is one implementation in `cursor_gui_scan.py` - `_pids_for(pgrep_pattern)`, `_detect_open_workspace_hashes(pids_fn, cache, ttl)` (lsof + the shared `workspaceStorage/<hash>/state.vscdb` regex, TTL-cached in the *caller-owned* cache dict so the two editors keep independent windows; `pids_fn` is a callable, not a list, so a warm cache skips pgrep and tests can still monkeypatch the per-editor `_*_pids` seam), and `_resolve_open_workspace(ws_hash, workspace_storage)` (workspace.json→folder). Each scanner keeps thin zero-arg wrappers (`_cursor_pids`/`_open_workspace_hashes`/`_workspace_for_hash`) that the tests monkeypatch. `vscode_copilot_scan` imports the three helpers (alongside the already-shared `_query_ro`/`_db_signature`/`_uri_to_path`/`_branch_for`). (2) **Per-editor presentation + actions** live in the `_EDITOR_PROFILES` table (`_EditorRowProfile` dataclass, keyed by `row_type`) in `table_builder_mixin.py`: `cli_label`/`app_label`/`chat_noun`/the two close tooltips, plus `on_close_full`/`on_close_server`/`on_jump` adapter callables that take the window first and reshape to each handler's real signature (the VS Code adapters drop the folder a Cursor close needs). `_build_editor_gui_row()` reads one profile and dispatches uniformly instead of branching on `is_vscode`; adding a third editor is a new table entry plus a new scanner, not new branches scattered through the builder.

## Cursor Agent tabs

**Why it's read-only (but tab-level jump works).** Cursor's Agent tabs live entirely inside its Electron app - no PTY/socket/public API to *drive* them (so no queueing/sending), and Cursor exposes **nothing clickable to macOS Accessibility** (verified live: 0 AX windows, `AXManualAccessibility` unsupported, the window collapses to nested empty `AXGroup`s). But *focusing* a tab is possible via Cursor's own command registry: `focus_cursor_window(folder, composer_id)` raises the window through the System Events AX bridge (`AXRaise` by window title - Cursor has no usable AppleScript dictionary, but System Events can read native window titles = workspace folder names; the raise itself is the shared `_raise_editor_window()` helper, also used for VS Code), then writes `focusComposer:<composer_id>` to `~/.leap-terminal-request`. The **Leap extension** (`src/leap/vscode-extension/extension.js`, the same `.vsix` ships to both editors) picks that up in the now-foreground window and calls `vscode.commands.executeCommand('composer.openComposer', composerId)` - the **bare id string** is essential: that reaches Cursor's `openComposerImpl` fast-path `selectedComposerIds.includes(id) -> showAndFocus(id)`, the actual visible tab switch (passing an object skips it, which is why `glass.openAgentById` resolved but never switched). Fallbacks `composer.openComposerFromNotification {composerId}` then `glass.openAgentById` cover builds without that command. Confirmed working live. Best-effort: if the extension isn't loaded or a Cursor build gates the command, the window-level raise still happened. The Cursor-only handlers are strictly gated via `isCursor()`, and the VS Code-only handlers are gated on `!isCursor()` - either way the wrong editor's window leaves the shared request file untouched rather than consuming it.

**Data source (all on-disk SQLite, undocumented/version-fragile -> every read is defensive).** `scan_open_cursor_agents()` joins three stores under `~/Library/Application Support/Cursor/User/`: (1) `workspaceStorage/<hash>/workspace.json` -> `{"folder": "file://.../<project>"}`; (2) that workspace's `state.vscdb` -> `ItemTable['composer.composerData'].selectedComposerIds` (the tabs **open in that window**); (3) global `globalStorage/state.vscdb` -> `cursorDiskKV['composerData:<id>']` per tab (`name`, `status`, `generatingBubbleIds`, `hasUnreadMessages`). **Which workspaces are open** is detected from the `state.vscdb` file handles Cursor holds open (`lsof -p <cursor pids>` -> `workspaceStorage/<hash>/state.vscdb`), TTL-cached. This is **frontmost-independent**: System Events / AX only enumerate Cursor's windows when it's the active app, so a window-title approach made the rows vanish whenever you looked away from Cursor. Reads use a read-only WAL connection (`?mode=ro` + `PRAGMA query_only`) with a copy-to-temp fallback, cached by db mtime signature.

**Status mapping** (`cursor_gui_scan._derive_status`, derived from persisted fields - Cursor's richer in-memory `activityState` isn't on disk): `generatingBubbleIds` non-empty OR `status == 'generating'` -> `running`; else `hasUnreadMessages` -> `unread`; else `idle`. Two deliberate, code-verified choices: (1) **no "Aborted"** - Cursor writes `status='aborted'` whenever a generation *ends* (`updateComposer(h, {chatGenerationUUID: undefined, status:'aborted'})`), including normal completion, so it maps to `idle` not a scary "Aborted"; (2) **`unread`, not "replied"** - `hasUnreadMessages` is Cursor's *manual* "Mark as unread" flag (cleared on view), not an auto "agent replied" signal. Status lags the live GUI by a poll tick (disk-flushed).

## VS Code Copilot Chat sessions

**Data source.** `scan_open_vscode_copilot_sessions()` (in `vscode_copilot_scan.py`) reads per-workspace stores under `~/Library/Application Support/Code/User/workspaceStorage/<hash>/`: `workspace.json` (same format as Cursor - both are VS Code at heart), and `state.vscdb` -> `ItemTable['chat.ChatSessionStore.index']` - a JSON index of every chat session (`title`, `lastMessageDate`, `isEmpty`, `isExternal`, `timing`, `lastResponseState`). **Verified live against a real Copilot generation**: VS Code rewrites the index within ~1s of a request starting and again at completion, so it's fresh enough for a status column. Open workspaces are detected exactly like Cursor (lsof on VS Code's pids, matched on `'Visual Studio Code.app'` so Cursor and Insiders are excluded). The last user prompt comes from `chatSessions/<sessionId>.jsonl` - an op-log (`kind` 0 = compacted full state with `v.requests`; later lines set `k == ['requests']` or `['requests', <idx>]`); the newest request list's last `message.text` wins, signature-cached. Shared low-level helpers (`_query_ro`, `_db_signature`, `_uri_to_path`, `_branch_for`) are imported from `cursor_gui_scan`.

**Status mapping** (`vscode_copilot_scan._derive_status`, from `lastResponseState` - the last request's response state; mapping verified empirically AND against VS Code's own index->status function which maps `{1,2}→Completed, 3→Failed, 0→InProgress, 4→NeedsInput` for historical entries): `2` -> `running` ('Running'), `4` -> `unread` ('Needs input' - the attention color), `3` -> `idle` with text 'Failed', anything else -> `idle`.

**Visibility (no "open tabs" notion to mirror).** VS Code keeps every chat ever in the store, so the visible set is: not `isEmpty` (VS Code pre-creates blank "New Chat" entries), not `isExternal` (cloud / other-provider sessions - the Agent Sessions view lists e.g. Claude Code there, which would double-count Leap's own rows), and then *generating* OR active within `RECENT_WINDOW_MS` (48h) OR PR-tracked (`keep_ids` - passed from the worker as the tracked `vscode-gui:` tags, so tracking survives age-out like a tracked-but-closed Cursor tab) - minus *hidden* sessions. **Hide instead of close**: VS Code has no chat tab to close, so the X buttons hide the row - `_hide_vscode_session_id()` persists `{session_id: dismissed_at_ms}` into the `vscode_gui_hidden` monitor pref (30-day-pruned), and the scan skips a hidden session until its `lastMessageDate` is NEWER than the dismiss time (a new user prompt auto-unhides - the moral equivalent of reopening a closed tab; `lastMessageDate` is the last *user* message time, verified live, so a dismissal mid-generation stays dismissed). The worker snapshots `vscode_hidden` / `vscode_keep_ids` on the GUI thread at construction.

**Jump and rename (both verified live).** `focus_vscode_chat_session(folder, session_id)` raises the window (shared `_raise_editor_window`, System Events process name `"Code"`), settles 0.3s, then writes `focusChatSession:<id>`; the extension (VS Code-gated) builds `vscode-chat-session://local/<base64url(sessionId)>` (VS Code registers a chat editor for that scheme) and runs `vscode.open` - the session opens as an **editor tab** (VS Code's native presentation for a session by id, not a selection in the SESSIONS list). Rename: unlike Cursor, VS Code's rename IS drivable - `rename_vscode_chat_session()` writes `renameChatSession:<id>` and the extension invokes `agentSession.rename` with the marshalled arg `{$mid: 25, session: {resource: <uri>}}` (a structural check on the main-thread side); VS Code shows its own input box pre-filled with the current title and persists via `setChatSessionTitle`, which lands in the store index -> the monitor row label follows on the next scan. The Tag-cell right-click menu on a VS Code row therefore has "Rename chat in VS Code..." in addition to the Leap-local alias actions.

## Integration design (both editors)

Rows are kept in `MonitorWindow._cursor_gui_rows` (holds BOTH editors' rows despite the name, as does `_cursor_row_cache`), **never** in `self.sessions` outside the render, so they bypass the server-centric paths that read the leap-only `self.sessions` (pinned-session auto-pin, sleep guard, dock-badge notifications). `_update_table` builds `combined = self.sessions + _cursor_gui_rows`, **interleaves it by `row_order`** (an editor-GUI tag joins `row_order` like any leap tag, so these rows can be **dragged and reordered exactly like regular rows** and remember their slot by chat id across close/reopen - new chats append at the end), renders against `combined`, then restores `self.sessions` to the leap-only list in `finally`. Drag-drop and the status-fire click handler are **tag-based** (via the table's `_row_tags` property, not `self.sessions` indices) so they work regardless of where a row lands; the drag payload carries the dragged row's *tag* (not its index) so a mid-drag table rebuild can't remap it (`_reorder_tags_for_drag` is the pure, unit-tested reorder). `_build_editor_gui_row()` paints all columns itself (no cell-cache) and the Tag cell is **alias-aware**: right-click -> Set/Rename/Remove alias gives a custom Leap-side label persisted by chat id, independent of the editor. The chat's own name (Cursor's "Rename Chat" -> `composerData.name`; VS Code's session title / custom title) is the default label and flows in automatically.

**Two close buttons (mirrors regular-row semantics).** The leftmost "×" (`_close_cursor_tab_and_untrack` / `_hide_vscode_chat_and_untrack`) stops PR tracking AND closes the Agent tab (Cursor) / hides the chat row (VS Code), so the row goes away entirely - the analog of a normal row's delete-X. The Server-cell "×" next to "Open" (`_close_cursor_tab` / `_hide_vscode_chat`) closes/hides ONLY the chat and leaves tracking intact - the analog of a normal row's close-server-X that keeps a PR-tracked row alive as a *dead* row. For this to work, `_reconcile_cursor_gui_rows()` (called from `_on_sessions_refreshed`) caches each live row in `_cursor_row_cache` and, on the next scan, **synthesizes a `_tab_closed` row** (`status_text='○  Tab closed'` for Cursor, `'○  Chat hidden'` for VS Code) for every tracked editor-GUI tag the scan no longer returns - so a tracked chat that's closed/hidden via the Server-X stays in the table (and keeps being PR-polled, since the synthesized row carries `project_path`/`branch` and is still in `_tracked_tags`). The synthesized row hides its Server-X (nothing left to close) and its "Open" reopens the chat (best-effort, via the id-based command - works for hidden VS Code chats too, since the session still exists in VS Code). `_untrack_cursor_pr` drops the cache entry so an untracked closed chat isn't re-synthesized. The stale-prune keeps any tag that's live OR synthesized (tracked-closed), and only cleans up editor-GUI PR state for tags that are neither.
